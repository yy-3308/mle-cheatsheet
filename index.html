Foundations
Core ML Concepts — What They Actually Mean

What is a Model?

A function that maps inputs to predictions
You feed it features (e.g. age, city, spend) and it outputs a prediction (e.g. "will churn: 0.83"). Training = adjusting internal numbers (weights) until predictions are close to real answers.
f(X) → ŷ
X = features  |  ŷ = predicted output
y = actual label (ground truth)

What is Bias?

How wrong your model is on average — systematic error
Bias measures how far off your predictions are from the truth, averaged across many predictions. A model predicting "hotel price = $150" when average is $200 has high bias — it's consistently off in the same direction.
Bias = E[ŷ] − y

E[ŷ] = average prediction across datasets
y = true value

High bias → predictions consistently wrong
Low bias → predictions right on average
In practice: compare training error to baseline. High train error = high bias.

What is Variance?

How much predictions change when training data changes
Train the same model on two slightly different datasets. If predictions change a lot — high variance. The model memorised the training data instead of learning general patterns.
Variance = E[(ŷ − E[ŷ])²]

= spread of predictions around their own mean

High variance → train error low, test error high
Low variance → predictions stable across datasets
In practice: big gap between train error and test error = high variance.

What is a Loss Function?

A score measuring how wrong the model is right now
Training = repeatedly asking "how wrong am I?" and adjusting. The loss function is the "how wrong" calculator. Lower is better. The model adjusts weights to minimise it.
MSE (regression):
Loss = (1/n) Σ (y − ŷ)²

Cross-entropy (classification):
Loss = −Σ y·log(ŷ)

MAE: Loss = (1/n) Σ |y − ŷ|
XGBoost uses log loss by default for classification.

What is Gradient Descent?

The algorithm that actually trains a model
Imagine the loss as a hilly landscape. Gradient descent finds the downhill direction and takes a small step that way. Repeat millions of times → reach the lowest point (minimum loss).
w = w − α · ∂Loss/∂w

w = weight (what we're adjusting)
α = learning rate (step size)
∂Loss/∂w = gradient (which direction is downhill)
Too high α → overshoots. Too low → too slow. Tree-based models (XGBoost) use gradient descent internally too.

Parameters vs Hyperparameters

Learned automatically vs set by you before training
Parameters — the model learns these from data. You never set them manually.
Parameters: weights, coefficients, tree split values
→ learned during training via gradient descent
Hyperparameters — you set these before training. They control how the model learns.
Hyperparameters: learning rate, max_depth,
n_estimators, regularization strength (C, lambda)
→ tuned via cross-validation (GridSearch, Optuna)

Confusion Matrix — TP / FP / TN / FN

The foundation of all classification metrics
Every classification prediction falls into one of 4 buckets:
Predicted: YES | Predicted: NO
─────────────────────────────
Actual YES │ TP (hit) │ FN (miss)
Actual NO │ FP (false │ TN (correct
│ alarm) │ reject)
TP = model said yes, was right
FP = model said yes, was wrong (false alarm)
FN = model said no, was wrong (missed it)
TN = model said no, was right
Precision = TP / (TP + FP)
Recall = TP / (TP + FN)
Accuracy = (TP + TN) / total

What is Regularization?

A penalty added to the loss to prevent overfitting
Without regularization, models learn to memorise training data (huge weights, overfits). Regularization adds a penalty for large weights — forcing the model to stay simpler and generalise better.
L1 (Lasso): Loss + λ·Σ|w|
→ drives small weights to exactly 0
→ automatic feature selection

L2 (Ridge): Loss + λ·Σw²
→ shrinks all weights toward 0
→ keeps all features, just smaller

λ = regularization strength (hyperparameter)

Train / Validation / Test Split

Why you need 3 separate data buckets
Training set — model learns from this (fits parameters)
Validation set — you tune hyperparameters here. Never used to fit weights.
Test set — locked away until the very end. One-time final evaluation. Gives true estimate of real-world performance.
Typical split: 70% train / 15% val / 15% test

Why not just train + test?
If you tune hyperparams on test set → you've
leaked test info → overly optimistic results.

What is Overfitting vs Underfitting?

The two ways a model can fail
Underfitting (high bias) — model is too simple. Can't even learn the training data. Train error is high.
Overfitting (high variance) — model memorised training data. Works perfectly on train, fails on new data. Gap between train and test error is large.
Underfit: train error HIGH, test error HIGH
Good fit: train error LOW, test error LOW
Overfit: train error LOW, test error HIGH

↑ model complexity → bias↓ variance↑
↓ model complexity → bias↑ variance↓

What is a Feature?

An input variable the model uses to make predictions
Raw data needs to be turned into numbers — that's feature engineering. Features are the columns in your training table.
Raw: "user checked in on Friday, spent $320"

Features: day_of_week=5, spend=320,
is_weekend=0, log_spend=5.77

Label (y): did_churn=1
Feature importance: XGBoost can rank which features matter most — useful for explaining the model to stakeholders.

What is Gradient Boosting / XGBoost?

An ensemble of weak trees that correct each other's errors
Build one small decision tree. It makes mistakes. Build a second tree to predict those mistakes. Add them together. Repeat 100–1000 times. Each tree "boosts" the previous one.
Final prediction = tree₁ + tree₂ + ... + treeₙ

Each treeₙ learns: residuals from previous trees
(i.e. where did we go wrong?)

XGBoost = optimised gradient boosting
with L1/L2 regularization built in
Why it wins on tabular data: handles mixed types, missing values, non-linear patterns, automatically.

ML Fundamentals (15-min quiz)

Bias vs Variance

  • High bias = model too simple → underfits → high train & test error
  • High variance = model too complex → overfits → low train, high test error
  • Sweet spot: tune complexity until train ≈ test error (both low)
  • Fix bias: more features, more complex model, less regularization
  • Fix variance: more data, regularization, simpler model, ensemble

Overfitting — Fixes

  • More training data — gold standard when available
  • L1 (Lasso) — drives weights to zero, feature selection
  • L2 (Ridge) — shrinks weights, keeps all features
  • Dropout — randomly zeros neurons during training (NNs)
  • Early stopping — stop when validation loss plateaus
  • Simpler model — fewer layers/trees/features
  • Cross-validation — detect overfit before deployment

Imbalanced Data

  • Accuracy is useless — 99% accurate by always predicting majority
  • SMOTE — synthetically oversample minority class
  • class_weight='balanced' — penalize majority errors more
  • Undersample — randomly drop majority rows
  • Threshold tuning — lower predict_proba threshold for minority
  • Use: Recall, ROC-AUC, F1, PR-AUC — not accuracy

Evaluation Metrics

  • Accuracy — (TP+TN)/total. Misleading if imbalanced.
  • Precision — TP/(TP+FP). "Of positives I called, how many right?"
  • Recall — TP/(TP+FN). "Of actual positives, how many caught?"
  • F1 — harmonic mean of precision & recall. Balances both.
  • ROC-AUC — rank quality across all thresholds. Needs predict_proba()
  • MAE — mean absolute error. Robust to outliers.
  • RMSE — penalizes large errors more. Sensitive to outliers.
  • R² — % variance explained. 1.0 = perfect, 0 = baseline mean.

When to Use Which Metric

ScenarioUse
Fraud / cancer detectionRecall + ROC-AUC
Spam filterPrecision
Imbalanced classesF1, PR-AUC
Ranking / recommendationsNDCG, MAP
Price predictionRMSE / MAE
Any regressionRMSE + R²

Model Selection

  • Tabular data → XGBoost / LightGBM (almost always best)
  • Linear relationships → Logistic / Linear Regression (fast, interpretable)
  • Text / images / sequences → Neural networks / Transformers
  • Small data, need interpretability → Decision Tree, Logistic Regression
  • Ensemble → Random Forest (bagging), XGBoost (boosting)
  • Clustering → K-Means (known k), DBSCAN (unknown k / noise)

Model Selection — Key Q&A

Why XGBoost over Random Forest?
Boosting = sequential, each tree corrects prior errors. Usually better accuracy. RF = parallel bagging, better when you want speed/simplicity.
When NOT to use a neural network?
Small tabular data — overfits easily. XGBoost wins. NNs shine on unstructured data (text, images) with lots of data.
How to pick between L1 and L2?
L1 if you suspect many features are irrelevant (want sparse model). L2 if all features matter and you just want smaller weights.

Cross-Validation

  • K-Fold — split into k folds, train on k-1, test on 1, rotate. Average score.
  • Stratified K-Fold — preserve class ratio in each fold. Use for imbalanced.
  • Time-series → TimeSeriesSplit — never use future data to predict past!
  • Rule: tune hyperparams on validation, report final score on held-out test set

Feature Engineering Recap

  • Encoding: binary → map(), 3+ categories → pd.get_dummies()
  • Scaling: StandardScaler (z-score), MinMaxScaler (0–1)
  • Nulls: fill mean/median/mode, or flag with binary column
  • Binning: pd.cut() for equal-width, pd.qcut() for equal-frequency
  • Data leakage: never use future info or target-correlated features created after the label
  • High cardinality: target encoding, frequency encoding, or embedding

ML Pipeline Pattern

from pyspark.ml import Pipeline
from pyspark.ml.feature import (
    StringIndexer, OneHotEncoder,
    VectorAssembler, StandardScaler
)
from pyspark.ml.classification import GBTClassifier

indexer = StringIndexer(inputCol="city", outputCol="city_idx")
encoder = OneHotEncoder(inputCol="city_idx", outputCol="city_vec")
assembler = VectorAssembler(
    inputCols=["age", "city_vec", "spend"],
    outputCol="features"
)
scaler = StandardScaler(inputCol="features", outputCol="scaled")
model = GBTClassifier(featuresCol="scaled", labelCol="label")

pipeline = Pipeline(stages=[indexer, encoder, assembler, scaler, model])
fitted = pipeline.fit(train_df)
preds = fitted.transform(test_df)

Handling Skew

  • Identify: df.groupBy("key").count().orderBy("count", asc=False)
  • Repartition: df.repartition(200, "user_id") — distributes evenly
  • Broadcast join: for small lookup tables — avoids shuffle entirely
  • Salting: add random prefix to hot key, join on salted key, then aggregate
  • AQE: Spark 3+ adaptive query execution handles skew automatically

Feature Store

  • What: centralized store for computed features, shared across models
  • Online store: low-latency lookup at serving time (Redis, DynamoDB)
  • Offline store: historical features for training (S3, BigQuery)
  • Point-in-time correctness: prevents leakage — features at label time only
  • Tool: Feast (open source), Tecton, Vertex AI Feature Store
  • DE angle: it's a data pipeline + serving layer — your territory

Model Registry & MLOps

  • Model Registry: version control for models (MLflow, SageMaker Registry)
  • Tracks: metrics, params, artifacts, lineage, stage (Staging → Production)
  • CI/CD for ML: retrain on schedule or data drift trigger → auto-eval → promote
  • A/B testing: route % traffic to challenger model, compare metrics
  • Shadow mode: run new model in parallel, log predictions, don't serve yet

  • Latency vs throughput: REST = low latency, batch = high throughput
  • Caching: cache predictions for identical feature vectors

Monitoring & Data Drift

  • Data drift: input distribution shifts (user behavior changes, new markets)
  • Concept drift: relationship between features and label changes
  • Detect: PSI (Population Stability Index), KS test, Jensen-Shannon divergence
  • Model decay: metrics drop over time → trigger retraining
  • Monitoring stack: Evidently, WhyLogs, Grafana + custom metrics
  • Business metrics: always monitor click-through, conversion — not just AUC

Retraining Strategy

  • Scheduled: retrain weekly/monthly (simple, predictable)
  • Triggered: retrain when drift detected or metric drops threshold
  • Continuous: stream new data, update model incrementally (online learning)
  • Full retrain vs fine-tune: full = safer, fine-tune = faster for stable distributions
  • Training data strategy: sliding window (recent data only) vs full history

"What would you do differently?" Framing

  • 1. Data quality first — validation, null checks, schema enforcement at ingest
  • 2. Feature versioning — unversioned features = silent regressions
  • 3. Monitor earlier — don't wait for user complaints to detect drift
  • 4. Reproducibility — log data snapshot + code version + seed used for every run
  • 5. Business alignment — always tie model metric to business KPI from day 1

Scalability Principles

  • Horizontal scaling: add more nodes (Spark executors, API replicas)
  • Partitioning: partition by time or user_id to parallelize processing
  • Caching: Redis for hot lookup features, CDN for model artifacts
  • Async processing: return job_id immediately, process in background
  • Rate limiting: protect serving endpoints from traffic spikes
  • Idempotency: safe to retry failed jobs without double-counting

System Design Interview — Framework

  • 1. Requirements: scale (QPS, data volume), latency, consistency needs
  • 2. High-level design: draw the boxes (ingest → process → store → serve)
  • 3. Data model: schema, partitioning strategy, indexes
  • 4. Deep dive: bottlenecks, failure modes, trade-offs
  • 5. Monitoring: what metrics to watch, alerts
  • Tip: Always estimate scale first before choosing tech. 1K QPS ≠ 1M QPS.

Design a fraud detection system

Coding — Key Patterns to Know

# Sliding window (common in system design coding)
def max_sum_subarray(nums, k):
    window_sum = sum(nums[:k])
    max_sum = window_sum
    for i in range(k, len(nums)):
        window_sum += nums[i] - nums[i-k]
        max_sum = max(max_sum, window_sum)
    return max_sum

# Two pointers
def find_pair_sum(arr, target):
    arr.sort()
    l, r = 0, len(arr)-1
    while l < r:
        s = arr[l] + arr[r]
        if s == target: return True
        elif s < target: l += 1
        else: r -= 1
    return False
